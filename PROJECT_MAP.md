# PROJECT_MAP — CarbonSense

> Comprehensive file-by-file map of the CarbonSense codebase, built so a coding agent (Codex) can understand the entire system at a glance.
>
> **Hackathon**: Hack2Skill Prompt Wars
> **App**: CarbonSense — "Duolingo for Climate Action"
> **Stack**: Vite + React 19 + TanStack Start frontend (port 5173), Express 5 + TypeScript backend (port 3001), Supabase (Postgres + Auth), Google Gemini 2.5 Flash, Plaid Sandbox.

---

## 1. High-Level Architecture

```
                       Browser (Vite dev :5173, prod via Cloudflare/Nitro)
                       │
                       │  HTML / React 19 / SSR via TanStack Start
                       │  Supabase JS client (auth)
                       │  axios  →  /api/*  (proxied in dev to :3001)
                       ▼
        ┌─────────────────────────────────────────────┐
        │  carbonsense-web/   (Vite + TanStack Start) │
        │  src/routes/*, src/components/*,            │
        │  src/lib/api.ts, src/hooks/useApi.ts        │
        │  Copilot FAB/Panel, StickyHeader, AppGate   │
        └──────────────────┬──────────────────────────┘
                           │  /api/*  (Bearer JWT from Supabase)
                           ▼
        ┌─────────────────────────────────────────────┐
        │  carbonsense-api/   (Express 5 + TS :3001)  │
        │  middleware/auth.ts (Supabase getUser)      │
        │  routes/*  →  controllers/*  →  services/*  │
        │  ai.service (Gemini 2.5 Flash)              │
        │  plaid.service (sandbox/dev/prod Plaid)     │
        │  ioredis (Upstash) — leaderboard cache      │
        └──────────────────┬──────────────────────────┘
                           │
              ┌────────────┴────────────┐
              ▼                         ▼
   ┌────────────────────┐   ┌──────────────────────────┐
   │ Supabase Postgres  │   │ External APIs            │
   │  + Auth (JWT)      │   │  • Plaid  (banking)      │
   │  + RLS per table   │   │  • Gemini 2.5 Flash (AI) │
   │  10 tables         │   │  • Stripe (configured)   │
   │  000 + 001 SQL     │   │  • Upstash Redis         │
   └────────────────────┘   └──────────────────────────┘
```

### Run modes

| Mode | Entry | Notes |
| --- | --- | --- |
| Single-process dev | `package.json` → `npm run dev` → `node scripts/dev-supervisor.js` | Port-aware supervisor; restarts on port collision |
| Concurrent dev | `npm run dev:raw` (`concurrently`) | Falls back if supervisor unavailable |
| API only | `npm run dev:api` → `cd carbonsense-api && npm run dev` | `tsx watch src/index.ts` |
| Web only | `npm run dev:web` → `cd carbonsense-web && npm run dev` | Vite dev server with `/api` proxy → `localhost:3001` |
| Production | Backend → Railway (`Procfile` + `railway.json`); Frontend → Vercel/CF | `vite build` → nitro bundle |

## 7. Cross-Cutting Concerns

### 7.1 Auth
- Frontend uses **Supabase JS client** (email/password + Google OAuth). Tokens stored in `localStorage` with Supabase's auto-managed session.
- `attachSupabaseAuth` TanStack middleware attaches `Bearer` token to every `createServerFn` request.
- `useAuthListener` hydrates `useAuthStore` (Zustand) with user metadata, `onboarding_complete`, streak, level.
- Backend `requireAuth` calls `supabase.auth.getUser(token)` and sets `req.user`. Throws 401 `AUTH_TOKEN_EXPIRED` vs `AUTH_TOKEN_INVALID` based on error message text.
- 401 in axios response → `supabase.auth.signOut()` + redirect `/login` (handled in `lib/api.ts` interceptor).

### 7.2 Carbon math
- **Per-merchant factors** (140 entries, 0.05–2.31 kg/USD) — fuzzy matched with Levenshtein + token overlap, threshold 0.82.
- **Per-category factors** (Plaid hierarchy, 50+ entries) — fallback after merchant match.
- **AI classification** — Gemini prompted for JSON `{carbon_category, emission_factor_per_dollar, reasoning}`. Zod-validated.
- **Confidence scores**: merchant=0.85, category=0.65, AI=0.50.
- **Carbon Age**: `max(0, round(bioAge + (annualTons - target) * 2))` where target = 4 tons.
- **Estimated data fallback** — `getDashboard` and `getComparison` synthesize weekly estimates from `onboarding_data.category_breakdown` or quiz answers when no live transactions exist.

### 7.3 Gamification
- 10-level system (0–5500 XP).
- Auto level-up in `addXP`. Milestone bonuses (7/14/30/60/100/365 day streaks) added by `incrementStreak`.
- Achievements auto-awarded by `checkAchievements` (called from `completeChallenge`) based on streak_count / challenges_completed / carbon_saved / level thresholds.

### 7.4 Streaks
- IST-based dates (5.5h offset applied everywhere via `utils/date.ts`).
- One streak per day (only first completion counts; subsequent same-day completes are no-ops).
- Streak freeze auto-regenerates every 7 days since signup.
- Daily cron (`/api/admin/run-daily-jobs`) rolls yesterday and uses freeze if user missed a day.

### 7.5 Plaid security
- AES-256-GCM token encryption with key derived from `JWT_SECRET` (SHA-256). Stored as `iv.authTag.ciphertext` in base64url.
- `disconnectBank` flips `status` to `disconnected` and soft-removes transactions (`is_removed: true`).
- Webhook handler syncs connections when `TRANSACTIONS/SYNC_UPDATES_AVAILABLE` arrives.

### 7.6 AI Copilot
- Single conversation per user (latest `copilot_conversations` row); messages capped at 50.
- System prompt built from user profile + monthly summary + recent challenges + equivalencies.
- Two Gemini calls per message: chat response + follow-up suggestions.
- Rate-limited to 20 req / 15 min via `aiRateLimit`.

### 7.7 Teams
- 8-char base64url invite codes (regenerated on collision, max 5 attempts).
- Leaderboard cached in Redis (`team:<id>:leaderboard:<period>`), TTL 1h, invalidated on join + on `updateTeamStats`.
- Anonymized non-admin display names ("Member #N").

### 7.8 SSR / error handling
- Custom `server.ts` SSR entry translates h3-swallowed 500 `{"unhandled":true,"message":"HTTPError"}` responses into a proper error page HTML.
- `error-capture.ts` records global errors with a 5s TTL so `server.ts` can log the real error.
- `errorMiddleware` in `start.ts` returns the error page HTML for non-`statusCode` exceptions.
- `<ErrorBoundary>` wraps each route in `__root.tsx` with `resetKey={pathname}` to reset on route change.

### 7.9 Responsive / UI
- Tailwind v4 (`@tailwindcss/vite`) + custom `responsive.css` (aurora orbs, glass panels, gradient borders, animations).
- Mobile breakpoint 768px (`useIsMobile`).
- Sidebar nav on `lg`, hamburger drawer on mobile (`framer-motion` slide-in).
- `UnitsProvider` for metric/imperial toggle (persists to `localStorage`).
- `react-hot-toast` for toasts (custom dark style in `__root.tsx`).

### 7.10 Styling/CSS structure
- `styles.css` (main) + `styles/responsive.css`. Both imported via `import "..."?url` in `__root.tsx` `head.links`.

### 9.2 Architecture / pattern reminders
- **Lovable-managed Vite config** — never add React/Vite/Tailwind/TS-paths plugins manually (duplicates break the dev server).
- **Lovable-managed Supabase middleware files** — `auth-middleware.ts`, `auth-attacher.ts`, `client.ts`, `client.server.ts`, `types.ts` are auto-generated. Don't hand-edit.
- **Backend uses service-role for everything.** RLS is bypassed. All authorization happens in controllers/services.
- **Date timezone is IST** for streaks, summaries, leaderboards, and challenge assignment. Backend code uses `utils/date.ts`.
- **Two parallel type systems**: backend `src/types/index.ts` uses snake_case (DB shape); frontend `src/types/index.ts` uses mixed camelCase (API shape). `lib/api.ts` normalizers handle the translation; if you add a new endpoint, add a normalizer.
- **`ErrorBoundary` + `AppGate`** wrap the whole tree in `__root.tsx` — every page is double-protected.
- **All hooks in `useApi.ts`** scope their query keys by `userId` via `useScopedUserId()`; switching users auto-resets.
- **`/api/admin/run-daily-jobs` is unauthenticated except for `x-admin-secret` header** — set `ADMIN_JOB_SECRET` to a strong random 32+ char string in prod.
- **Plaid `plaid_access_token` is encrypted at rest** (AES-256-GCM, key derived from `JWT_SECRET`). If you rotate `JWT_SECRET`, every stored connection becomes undecryptable.
- **Redis is optional in dev** — production must set `REDIS_URL`.

### 9.3 Things that work but are fragile
- `getDashboard` has 10 parallel queries and synthesizes missing data aggressively; any new dashboard field must be added here AND in the frontend's `Dashboard` interface.
- `carbon_summaries` are recomputed only by `runDailyJobs` (yesterday) or `syncTransactions` (per affected date). Calling `GET /carbon/dashboard` between two syncs may show stale "this_month" totals.
- `CORS` allowlist hard-codes the dev ports — production must set `FRONTEND_URL` to the deployed origin.
